# CS-CORE.3A Legacy Profile.small_group Consumer Inventory

## Purpose and Status

This is a post-CS-CORE.2B-A / post-CS-CORE.2C-B inventory of remaining runtime consumers that still use legacy `Profile.small_group`, `SmallGroup`, `District`, or legacy structure bridge fields after the completed membership-core switches. CS-CORE.3B later moved Bible Study v2 role and worship user pickers to membership-core filtering; BS-STRUCT.2A later made `BibleStudyMeetingAudienceScope` rows the V2 runtime source of truth for ordinary-member meeting visibility, `/studies/` / Today, and role/worship pickers, with zero-row V2 meetings failing closed and `BibleStudyMeeting.small_group` kept as mirror/display/backfill/history/idempotency compatibility only. BS-V2-MIRROR.1A later moved V2 member/staff display labels toward `anchor_unit` / meeting audience units, leaving `BibleStudyMeeting.small_group` as stored mirror/history/idempotency compatibility and fallback display for old/invalid rows. BS-V2-MIRROR.1B later stopped new generated/manual normal V2 meeting writes from setting `BibleStudyMeeting.small_group`; BS-V2-MIRROR.1C adds a dry-run-first guarded cleanup command for existing stored mirror values, clearing only rows that already have correct `generation_key`, `anchor_unit`, and exactly one matching active small-group audience row. BS-SERIES-SCOPE.1A later stopped normal app-level Bible Study schedule saves from writing legacy `BibleStudySeries.scope_type`, `ministry_context`, `district`, or `small_group`; BS-SERIES-SCOPE.1B adds the dry-run-first guarded `cleanup_bible_study_series_legacy_scope_fields` command for existing populated values, clearing only series rows that already have valid `BibleStudySeriesAudienceScope` rows. No cleanup runs automatically, unsafe/mismatched rows remain blocked for review, and this does not remove model fields or DB constraints. PRAYER-STRUCT.1A later moved ordinary group-prayer visibility to `PrayerRequest.structure_unit_at_post` + active primary `ChurchStructureMembership`, leaving `PrayerRequest.small_group_at_post` as legacy mirror/history only. Rows below mark those slices as complete. CS-CORE.3C later recorded the docs-only decision that Bible Study V2 is the active product path and legacy V1 `BibleStudySession` is a retirement candidate (`docs/LEGACY_BIBLE_STUDY_SESSION_RETIREMENT_DECISION.md`). CS-CORE.3D froze app-level V1 creation through `/studies/new/`; CS-CORE.3F froze remaining V1 app-level edit/delete/worship mutation routes; BS-V1-RETIRE.1A then retired V1 `BibleStudySession` app-level runtime entirely, so direct V1 detail/list access redirects for ordinary users and managers. Remaining V1 rows are pilot/archive data, not a protected production archive and not a membership-core migration target; BS-V1-PURGE.1A adds a guarded dry-run-first purge command, but no V1 rows are deleted unless staff explicitly run the double-confirmed apply command. CS-CORE.4F.1/4F.2 switched the group-progress visible roster and permission-fenced default selected group to membership-core, CS-CORE.4G.2 switched the ordinary-member group reflection **read path** to `structure_unit_at_post` + active primary `ChurchStructureMembership`, and CS-CORE.4G.3 switched the reflection **write path / form option-gating** to the same membership-core source; rows below reflect those switches. CS-CORE.2D-A added the structure-aware role-scope **foundation** (nullable `ChurchRoleAssignment.structure_unit`, read-only resolution helpers, and the `audit_structure_role_scopes` readiness command), and CS-CORE.2D-B then **switched the group-progress permission/access list** (`get_accessible_progress_groups()` / `can_view_group_progress_for()`) to structure-aware role scopes plus membership-core own-group resolution; `Profile.small_group` no longer grants any group-progress access. ROLE-RETIRE.1B then retired the legacy `ChurchRoleAssignment.district` / `small_group` role-scope **runtime** fallback, so scoped role runtime now requires an explicit `structure_unit`; those legacy role fields remain stored for admin/display/audit/backfill/rollback context only.

This inventory does not authorize migrations, source-of-truth changes, permission changes, schema changes, data changes, or runtime behavior changes. It is an audit snapshot to help plan small future slices.

## LEGACY-RETIRE.1A Retirement Readiness Foundation

LEGACY-RETIRE.1A adds the comprehensive read-only command `audit_legacy_structure_retirement_readiness` and the execution tracker `docs/LEGACY_STRUCTURE_RETIREMENT_EXECUTION_PLAN.md`. The command reports current data blockers across `Profile.small_group`, `SmallGroup`, `District`, `MinistryContext`, ServiceEvent legacy scope fields, Bible Study legacy fields / V1 sessions / generation bridges, reflection legacy snapshots, role legacy fields, and diagnostic/backfill tooling. It has `--verbose`, `--limit N`, and `--fail-on-blockers`; it has no `--apply`, writes no data, and changes no runtime behavior.

Use the new execution plan as the current checklist for legacy-field/table retirement sequencing. Keep this consumer inventory as the detailed historical/current-state map of which consumers are switched, stored-only, diagnostic-only, retired app runtime, or still legacy bridge/runtime.

## Migration Status (CS-CORE.2D-B; CLOSE.1 superseded)

**The Church Structure core migration is the current top priority and is being accelerated.** An earlier uncommitted CS-CORE.CLOSE.1 draft proposed "closing" the migration after CS-CORE.4G.1 and deferring the reflection read-path switch (citing demo/QA data as an insufficient gate); **that draft was superseded before commit** and is not current policy (`docs/CHURCH_STRUCTURE_CORE_MIGRATION_PLAN.md` Section 1A). Do not treat the remaining legacy consumers below as permanently deferred or as a "closed" set — they are sequenced remaining work, each its own approved slice. They are still intentionally retained **for now** as a compatibility layer and are not unfinished bugs.

CS-CORE.4G.2 switched the ordinary-member group reflection **read paths**, and CS-CORE.4G.3 switched the reflection **write path / form option-gating**, both to `structure_unit_at_post` + active primary `ChurchStructureMembership` (see the Current Switched Consumers table below).

Remaining legacy work (active, sequenced — not blocked on demo/QA data):

- **Legacy `ChurchRoleAssignment.district` / `small_group` role scope fields** — the **runtime** role-scope fallback is **retired (ROLE-RETIRE.1B)** and scoped-role validation is now structure-unit-native (ROLE-SCHEMA.1A): group-progress permission and new scoped-role validation both resolve a scoped role's structure scope through the explicit valid `structure_unit` **only**, and a non-global scoped assignment with no `structure_unit` fails closed/does not validate. **CS-CORE.2D-C added backfill tooling** (`backfill_structure_role_scopes`, dry-run by default, `--apply` to write) that populates `structure_unit` from these mapped legacy scopes on existing rows; a real-data gate (audit → dry-run → apply → re-audit) confirmed zero rows needed migration before retirement. The legacy fields are **not removed** — they remain stored/admin/display/audit/backfill/rollback context only and no longer grant runtime access or substitute for `structure_unit` validation. The audit (`audit_structure_role_scopes`) still derives a candidate unit from these legacy fields via the diagnostic-only `resolve_role_assignment_structure_unit_for_diagnostics()`. Remaining work is field-level retirement of these now-display-only fields.
- **Legacy structure parent/context links (`SmallGroup.district`, `District.ministry_context`)** — **LEGACY-OBJECT-LINKS.1A** adds the dry-run-first guarded `cleanup_legacy_structure_parent_links` command. These two legacy FKs duplicate the `ChurchStructureUnit.parent` hierarchy; the command nulls them only when the child unit's `parent` already equals the parent legacy object's mapped active unit (matching unit types, both active). The migrated runtime does not depend on them — ordinary visibility is audience rows + active primary `ChurchStructureMembership`, and the legacy-parity resolver `accounts.structure_selectors.resolve_units_to_small_groups()` matches eligible groups through hierarchy descendants (its `district` / `district__ministry_context` branches are redundant for exactly the cleared links). `BibleStudySeries.get_eligible_small_groups()` uses those legacy branches only as a coexistence fallback for series with legacy scope and no audience rows (currently zero). `ServiceEvent.ministry_context` is reported but **conservatively skipped** (editable "Host / Language Label", see the ServiceEvent section). The command deletes no rows, removes no fields, runs no schema migration, and switches no runtime source of truth. Re-seeding caution: `seed_church_structure_units` derives parents from these links when first building the tree, so after clearing them the hierarchy is authoritative in `ChurchStructureUnit.parent` and the seed command should not be re-applied expecting parent re-derivation.
- **Legacy `Profile.small_group` / `SmallGroup` / `District` / `MinistryContext`** — kept as the compatibility layer for now; retire per-table once no runtime consumer reads them. **CS-RETIRE.1A: no normal app-level flow writes `Profile.small_group` anymore.** After BS-V1-RETIRE.1A, `Profile.small_group` no longer grants V1 `BibleStudySession` app access either. The remaining `Profile.small_group` uses are read-only audit / shadow-comparison commands, Django Admin emergency/legacy maintenance, read-only staff/legacy display, and historical/backfill compatibility tooling. The profile page now shows the active primary `ChurchStructureMembership` as the confirmed group, and membership approval no longer mirrors into `Profile.small_group`. The field/model is **not** removed.
- **ServiceEvent zero-row legacy runtime fallback** — **retired in SE-RETIRE.1B** (no longer an active runtime consumer of `Profile.small_group`). After the SE-RETIRE.1A audit ran clean on production, `ServiceEvent.can_be_seen_by` was changed so an event with zero `ServiceEventAudienceScope` rows **fails closed** for ordinary users; it no longer reads `scope_type` / `district` / `small_group` or `Profile.small_group`. Manager/staff override is unchanged. SE-SCOPE.1A later stopped normal app-level ServiceEvent create/edit and recurring saves from writing `scope_type`, `district`, or `small_group`. SE-SCOPE.1B adds the guarded dry-run-first `cleanup_service_event_legacy_scope_fields` command for existing stored values, clearing only rows that already have `ServiceEventAudienceScope` rows and only after `--apply --confirm-service-event-legacy-scope-cleanup`. The legacy fields are **not** removed, cleanup does not run automatically, and any zero-row candidate remains blocked/preserved. `audit_service_event_fallback_retirement_readiness` remains as a standing guard/diagnostic (a zero-row event is now an invisible safety state), and `backfill_service_event_audience_scopes` still reads legacy fields for backfill/audit support. Legacy field/schema retirement is a separate decision.
- **Bible Study v2 legacy `SmallGroup` meeting-generation bridge** — kept for secondary idempotency, history, fallback display, diagnostics, and old-row compatibility. It is no longer an ordinary-member V2 visibility fallback after BS-STRUCT.2A, no longer the preferred member/staff display label source after BS-V2-MIRROR.1A, and no longer written for new generated/manual normal V2 meetings after BS-V2-MIRROR.1B. BS-V2-MIRROR.1C adds guarded dry-run-first cleanup for existing safe `BibleStudyMeeting.small_group` mirrors only; unsafe/mismatched rows remain review blockers and field/constraint removal is separate.
- **Legacy roles/capabilities and TeamAssignment / My Serving / ministry team membership** — kept separate; serving is never inferred from church-structure membership.

Already-switched consumers (ServiceEvent audience rows, Prayer group requests, Bible Study v2 meeting visibility / Today / role-worship pickers, group-progress roster and permission-fenced default group, the group-progress permission / accessible group list, and ordinary-member group reflection read **and write** paths) are listed in the next section.

## Non-Consumers / Do Not Migrate Casually

Do not migrate these casually as part of small-group cleanup:

- Permissions / capabilities: `ChurchRoleAssignment` and `accounts.permissions` are a role/capability system, not a membership-core visibility consumer.
- TeamAssignment / My Serving: serving assignments use `TeamMembership` and `TeamAssignmentMember`; do not infer serving roles from `ChurchStructureMembership`.
- MinistryTeam membership: ministry team membership remains a separate operational model.
- `BibleStudyMeetingRole`: treat as meeting-specific assignment metadata, not a ministry serving assignment or permission model.
- Reading progress and group progress: high privacy and permission risk; requires separate planning. CS-CORE.4F.1 switched only the group-progress visible roster (`member_rows`) to membership-core, and CS-CORE.4F.2 switched only the no-`?group=` default selected group to a permission-fenced membership-core candidate (used only when already inside the legacy-accessible list), both after zero-drift gate evidence; the group-progress permission and the accessible group list then switched to structure-aware role scopes plus membership-core own-group resolution in CS-CORE.2D-B (foundation CS-CORE.2D-A), so `Profile.small_group` no longer grants group-progress access.
- Reflection/comment privacy: high privacy risk, especially because `small_group_at_post` is a historical snapshot. The ordinary-member group **read path** is **switched in CS-CORE.4G.2** and the **write path / form option-gating** is **switched in CS-CORE.4G.3**, both to `structure_unit_at_post` + active primary `ChurchStructureMembership` (fail-closed). `structure_unit_at_post` is the canonical read/write snapshot; `small_group_at_post` is now an optional legacy compatibility mirror and is not removed. Existing group posts preserve their snapshots under Policy C — do not re-home them casually.
- Legacy `BibleStudySession`: CS-CORE.3C decided retirement over migration, BS-V1-RETIRE.1A retired V1 app-level runtime entirely, and BS-V1-PURGE.1A adds guarded dry-run-first cleanup tooling for pilot rows. Do not migrate V1 `BibleStudySession` visibility to membership-core or create a parallel V1 archive product; see `docs/LEGACY_BIBLE_STUDY_SESSION_RETIREMENT_DECISION.md`.
- ServiceEvent zero-row fallback: the **runtime** fallback is retired (SE-RETIRE.1B) — zero-row events fail closed for ordinary users. SE-SCOPE.1A stops normal app-level writes to legacy `scope_type` / `district` / `small_group`; SE-SCOPE.1B adds guarded cleanup tooling for existing values on events with audience rows, while zero-row candidates remain blocked and field-level retirement remains later.
- Legacy `Profile.small_group`: do not remove or hide until all consumers in this inventory are migrated or retired (compatibility layer retained for now).
