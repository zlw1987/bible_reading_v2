# CS-CORE.3A Legacy Profile.small_group Consumer Inventory

## Purpose and Status

This is a post-CS-CORE.2B-A / post-CS-CORE.2C-B inventory of remaining runtime consumers that still use legacy `Profile.small_group`, `SmallGroup`, `District`, or legacy structure bridge fields after the completed membership-core switches. CS-CORE.3B later moved Bible Study v2 role and worship user pickers to membership-core filtering; BS-STRUCT.2A later made `BibleStudyMeetingAudienceScope` rows the V2 runtime source of truth for ordinary-member meeting visibility, `/studies/` / Today, and role/worship pickers, with zero-row V2 meetings failing closed and `BibleStudyMeeting.small_group` kept as mirror/display/backfill/history/idempotency compatibility only. BS-V2-MIRROR.1A later moved V2 member/staff display labels toward `anchor_unit` / meeting audience units, leaving `BibleStudyMeeting.small_group` as stored mirror/history/idempotency compatibility and fallback display for old/invalid rows. BS-V2-MIRROR.1B later stopped new generated/manual normal V2 meeting writes from setting `BibleStudyMeeting.small_group`; BS-V2-MIRROR.1C adds a dry-run-first guarded cleanup command for existing stored mirror values, clearing only rows that already have correct `generation_key`, `anchor_unit`, and exactly one matching active small-group audience row. BS-SERIES-SCOPE.1A later stopped normal app-level Bible Study schedule saves from writing legacy `BibleStudySeries.scope_type`, `ministry_context`, `district`, or `small_group`; BS-SERIES-SCOPE.1B added the dry-run-first guarded `cleanup_bible_study_series_legacy_scope_fields` command for existing populated values; **BS-SERIES-FIELD-RETIRE.1A then removed those four legacy series scope fields** (migration `studies/0010`) and retired the cleanup command, leaving `BibleStudySeriesAudienceScope` rows as the sole schedule audience source and structure-unit-native generation (fail closed on zero rows). STRUCTURE-BRIDGE.1A adds the read-only `audit_bible_study_generation_bridge_retirement` command to inventory the remaining Bible Study `SmallGroup` bridge split: ordinary V2 visibility is already audience-row + membership-core, normal generation is already structure-unit-native, and remaining legacy dependencies are old-row idempotency, stored mirrors, fallback display, admin, and diagnostic/cleanup support. No cleanup runs automatically, unsafe/mismatched rows remain blocked for review, and this does not remove model fields or DB constraints. PRAYER-STRUCT.1A later moved ordinary group-prayer visibility to `PrayerRequest.structure_unit_at_post` + active primary `ChurchStructureMembership`, leaving `PrayerRequest.small_group_at_post` as legacy mirror/history only. Rows below mark those slices as complete. CS-CORE.3C later recorded the docs-only decision that Bible Study V2 is the active product path and legacy V1 `BibleStudySession` is a retirement candidate (`docs/LEGACY_BIBLE_STUDY_SESSION_RETIREMENT_DECISION.md`). CS-CORE.3D froze app-level V1 creation through `/studies/new/`; CS-CORE.3F froze remaining V1 app-level edit/delete/worship mutation routes; BS-V1-RETIRE.1A then retired V1 `BibleStudySession` app-level runtime entirely, so direct V1 detail/list access redirects for ordinary users and managers. Remaining V1 rows are pilot/archive data, not a protected production archive and not a membership-core migration target; BS-V1-PURGE.1A adds a guarded dry-run-first purge command, but no V1 rows are deleted unless staff explicitly run the double-confirmed apply command. CS-CORE.4F.1/4F.2 switched the group-progress visible roster and permission-fenced default selected group to membership-core, CS-CORE.4G.2 switched the ordinary-member group reflection **read path** to `structure_unit_at_post` + active primary `ChurchStructureMembership`, and CS-CORE.4G.3 switched the reflection **write path / form option-gating** to the same membership-core source; rows below reflect those switches. CS-CORE.2D-A added the structure-aware role-scope **foundation** (nullable `ChurchRoleAssignment.structure_unit`, read-only resolution helpers, and the `audit_structure_role_scopes` readiness command), and CS-CORE.2D-B then **switched the group-progress permission/access list** (`get_accessible_progress_groups()` / `can_view_group_progress_for()`) to structure-aware role scopes plus membership-core own-group resolution; `Profile.small_group` no longer grants any group-progress access. ROLE-RETIRE.1B then retired the legacy `ChurchRoleAssignment.district` / `small_group` role-scope **runtime** fallback, so scoped role runtime now requires an explicit `structure_unit`; ROLE-FIELD-RETIRE.1A then **removed** those legacy role fields entirely (migration `accounts/0011`). **(Superseded:** the fields this chronological paragraph describes as "kept as mirror" / "stored compatibility" / "legacy mirror/history" — including `BibleStudyMeeting.small_group` and `PrayerRequest.small_group_at_post` — have since been removed; see the superseded-state banner immediately below.)

This inventory does not authorize migrations, source-of-truth changes, permission changes, schema changes, data changes, or runtime behavior changes. It is an audit snapshot to help plan small future slices.

> **PROFILE-SG-FIELD-RETIRE.1A — `Profile.small_group` removed (migration `accounts/0012`).** After preflight audits confirmed zero populated values and no live runtime/app-write/display/admin dependency, the `Profile.small_group` model field was removed. `ChurchStructureMembership` is the canonical user belonging source for migrated runtime paths; normal app-level membership approval/profile flows do not write any legacy profile group field. The guarded profile cleanup command (`cleanup_profile_small_group`), the profile-vs-membership belonging drift audit (`audit_structure_belonging`), the membership backfill that sourced from the field (`backfill_church_structure_memberships`), and the group-progress legacy shadow comparison (`audit_group_progress_shadow`, plus the shadow functions in `reading.group_progress_shadow`) were retired with it. This did **not** remove the `SmallGroup`, `District`, or `MinistryContext` tables, and did not affect ServiceEvent, Prayer, Reflection, Role, Bible Study, TeamAssignment, or V1 schema. Only immutable historical migrations still name the field. The consumer rows below are therefore historical.
>
> **Also superseded since this inventory was written:** `BibleStudyMeeting.small_group` was removed in BS-MEETING-MIRROR.1A (migration `studies/0011`); the legacy `BibleStudySeries` scope fields in BS-SERIES-FIELD-RETIRE.1A (`studies/0010`); the `ChurchRoleAssignment.district` / `small_group` fields in ROLE-FIELD-RETIRE.1A (`accounts/0011`); the `ServiceEvent.scope_type` / `district` / `small_group` fields in SE-FIELD-RETIRE.1A (`events/0007`) and `ServiceEvent.ministry_context` in SERVICE-EVENT-CONTEXT.1C (`events/0008`); `PrayerRequest.small_group_at_post` in PRAYER-MIRROR.1D (`prayers/0004`); and `ReflectionComment.small_group_at_post` in REFLECTION-MIRROR.1H (`comments/0007`). The diagnostic/backfill commands named in the rows below — including `audit_bible_study_membership_readiness`, `audit_structure_belonging`, `backfill_church_structure_memberships`, and `audit_group_progress_shadow` — were retired with their respective fields and no longer exist. Body rows that describe any of these as live runtime/mirror/display/command consumers are historical. **BS-V1-ADMIN-RETIRE.1A** also retired the active V1 Django Admin surface (unregistered `BibleStudySessionAdmin` and the V1-only `BibleStudyGuideAdmin` / `BibleStudyWorshipSongAdmin` in `studies/admin.py`); this was admin-only (no V1 data deleted, no V1 model/table/field removed, purge not applied), so the schema-retirement audit now classifies V1 `BibleStudySession` as `blocked_by_diagnostic_tooling` rather than `blocked_by_display_or_admin`. The remaining legacy blockers are the `SmallGroup` / `District` / `MinistryContext` tables (with their bridge `church_structure_unit` FKs and admin/diagnostic surfaces) and V1 `BibleStudySession` cleanup (its remaining pilot/archive rows plus a later model/table/schema removal slice).

## LEGACY-RETIRE.1A Retirement Readiness Foundation

LEGACY-RETIRE.1A adds the comprehensive read-only command `audit_legacy_structure_retirement_readiness` and the execution tracker `docs/LEGACY_STRUCTURE_RETIREMENT_EXECUTION_PLAN.md`. The command reports current data blockers across `Profile.small_group`, `SmallGroup`, `District`, `MinistryContext`, ServiceEvent legacy scope fields, Bible Study legacy fields / V1 sessions / generation bridges, reflection legacy snapshots, role structure scope, and diagnostic/backfill tooling. It has `--verbose`, `--limit N`, and `--fail-on-blockers`; it has no `--apply`, writes no data, and changes no runtime behavior.

ROW-RETIRE.1A adds the read-only command
`audit_legacy_structure_object_row_retirement` for the remaining
`SmallGroup`, `District`, and `MinistryContext` object rows themselves. It
classifies live code consumers by category, verifies that ordinary-member
runtime visibility no longer depends on the legacy object rows, reports mapping
health (`unmapped_rows`, `inactive_mapping_units`, `wrong_type_mapping_units`),
and highlights `District #13 未分配小组` / `UNASSIGNED-GROUPS` as a special
custom-unit placeholder decision. It has `--verbose`, `--limit N`, and
`--fail-on-blockers`; it has no `--apply`, writes no data, changes no schema,
and changes no runtime behavior.

LEGACY-SCHEMA-PREP.1A adds the read-only command
`audit_legacy_structure_schema_retirement_readiness` for field/table retirement
planning. It inventories candidate legacy fields/tables across Accounts,
ServiceEvent, Bible Study, and Reflection with data counts and code-reference
categories, while keeping diagnostics, admin/display references, tests/fixtures,
and migration history separate from runtime blockers. It has `--verbose`,
`--limit N`, and `--fail-on-blockers`; it has no `--apply`, writes no data,
changes no schema, changes no runtime behavior, and does not approve field or
table removal. Cleanup tooling should remain until target DBs are audited after
any separately approved cleanup apply, and legacy object tables remain last.

## Migration Status (CS-CORE.2D-B; CLOSE.1 superseded)

**The Church Structure core migration is the current top priority and is being accelerated.** An earlier uncommitted CS-CORE.CLOSE.1 draft proposed "closing" the migration after CS-CORE.4G.1 and deferring the reflection read-path switch (citing demo/QA data as an insufficient gate); **that draft was superseded before commit** and is not current policy (`docs/CHURCH_STRUCTURE_CORE_MIGRATION_PLAN.md` Section 1A). Do not treat the remaining legacy consumers below as permanently deferred or as a "closed" set — they are sequenced remaining work, each its own approved slice. They are still intentionally retained **for now** as a compatibility layer and are not unfinished bugs.

CS-CORE.4G.2 switched the ordinary-member group reflection **read paths**, and CS-CORE.4G.3 switched the reflection **write path / form option-gating**, both to `structure_unit_at_post` + active primary `ChurchStructureMembership` (see the Current Switched Consumers table below).

Remaining legacy work (active, sequenced — not blocked on demo/QA data):

- **Legacy `ChurchRoleAssignment.district` / `small_group` role scope fields — REMOVED (ROLE-FIELD-RETIRE.1A).** The runtime role-scope fallback was already retired (ROLE-RETIRE.1B) and scoped-role validation made structure-unit-native (ROLE-SCHEMA.1A); group-progress permission and scoped-role validation resolve a scoped role's structure scope through the explicit valid `structure_unit` **only**, and a non-global scoped assignment with no `structure_unit` fails closed/does not validate. **ROLE-FIELD-RETIRE.1A then removed both legacy fields** (migration `accounts/0011`) after local/dev audit confirmed zero populated legacy role fields and zero scoped assignments missing `structure_unit`. The now-orphaned `backfill_structure_role_scopes` command and the diagnostic-only `resolve_role_assignment_structure_unit_for_diagnostics()` helper were retired with the fields; `audit_structure_role_scopes` now validates explicit `structure_unit` readiness only, and the schema-retirement audit classifies both removed fields as historical-only. This did not change permissions or group-progress access and did not affect `Profile.small_group`, `SmallGroup`, `District`, `MinistryContext`, ServiceEvent, Bible Study, Reflection, or Prayer schema.
- **Legacy structure parent/context links (`SmallGroup.district`, `District.ministry_context`)** — **LEGACY-OBJECT-LINKS.1A** adds the dry-run-first guarded `cleanup_legacy_structure_parent_links` command. These two legacy FKs duplicate the `ChurchStructureUnit.parent` hierarchy; the command nulls them only when the child unit's `parent` already equals the parent legacy object's mapped active unit (matching unit types, both active). The migrated runtime does not depend on them — ordinary visibility is audience rows + active primary `ChurchStructureMembership`, and the structure-mapping resolver `accounts.structure_selectors.resolve_units_to_small_groups()` matches eligible groups through hierarchy descendants mapped via `SmallGroup.church_structure_unit` only. **LEGACY-BRIDGE-RESOLVER-NARROW.1A removed the resolver's `district` / `district__ministry_context` fallback branches**, so the resolver no longer reads `SmallGroup.district` or `District.ministry_context`. `BibleStudySeries.get_eligible_small_groups()` resolves `BibleStudySeriesAudienceScope` rows only (BS-SERIES-FIELD-RETIRE.1A removed the legacy series scope fields, so there is no legacy fallback) and inherits the structure-mapping-only resolution. The command does not scan `ServiceEvent`: the legacy `ServiceEvent.ministry_context` display FK was **removed in SERVICE-EVENT-CONTEXT.1C** (migration `events/0008`). Host / Language display now uses display-only `ServiceEvent.host_language_unit`, then the audience-derived `ChurchStructureUnit.parent` fallback; the display-only backfill/cleanup tooling (`backfill_service_event_host_language_units`, `cleanup_service_event_ministry_context_labels`) was retired with the field. Re-seeding caution: `seed_church_structure_units` derives parents from these links when first building the tree, so after clearing them the hierarchy is authoritative in `ChurchStructureUnit.parent` and the seed command should not be re-applied expecting parent re-derivation.
- **Legacy `SmallGroup` / `District` / `MinistryContext` tables** — kept as the compatibility/mapping layer for now. `ChurchStructureUnit` is the canonical structure tree and `ChurchStructureMembership` is the canonical belonging source for migrated ordinary-member paths, so these legacy rows must not be used as ordinary-member visibility authority. After ROW-RETIRE.1A, the remaining `SmallGroup` / `District` / `MinistryContext` blockers are the object rows themselves plus bridge/admin/display/diagnostic/setup consumers, not app runtime access dependencies. `District #13 未分配小组` mapped to `UNASSIGNED-GROUPS` is a custom-unit placeholder/holding-bucket decision, not a real district to "fix" during inventory.
- **Legacy `Profile.small_group`** — **removed in PROFILE-SG-FIELD-RETIRE.1A** (migration `accounts/0012`). Belonging is active primary `ChurchStructureMembership`; the profile cleanup/belonging-drift/backfill/shadow tooling and the legacy profile selector helpers were retired with the field, and the profile page shows the active primary `ChurchStructureMembership` as the confirmed group. Only immutable historical migrations/docs may name the field. This did not remove the `SmallGroup` / `District` / `MinistryContext` tables.
- **ServiceEvent legacy scope fields & zero-row fallback** — the zero-row runtime fallback was **retired in SE-RETIRE.1B** (an event with zero `ServiceEventAudienceScope` rows **fails closed** for ordinary users; `scope_type` / `district` / `small_group` and `Profile.small_group` are not consulted; manager/staff override unchanged), SE-SCOPE.1A then stopped normal app-level writes, and **SE-FIELD-RETIRE.1A removed `scope_type` / `district` / `small_group`** (migration `events/0007`) after local/dev audit confirmed all events carry audience rows with zero populated legacy scope fields. The legacy-scope tooling (`cleanup_service_event_legacy_scope_fields`, `backfill_service_event_audience_scopes`, `audit_service_event_fallback_retirement_readiness`) was retired with the fields. Visibility is `ServiceEventAudienceScope` rows + active primary `ChurchStructureMembership`; only immutable historical migrations name the fields.
- **Bible Study v2 `BibleStudyMeeting.small_group` mirror** — **removed in BS-MEETING-MIRROR.1A** (migration `studies/0011`). It was already not an ordinary-member V2 visibility fallback after BS-STRUCT.2A, not the preferred display label source after BS-V2-MIRROR.1A, and not written for new generated/manual normal V2 meetings after BS-V2-MIRROR.1B; BS-V2-MIRROR.1C cleared safe stored mirror values. After preflight audits confirmed zero populated values and no live runtime/visibility/display/admin/generation dependency, the field was removed; V2 meeting visibility remains `BibleStudyMeetingAudienceScope` rows + active primary `ChurchStructureMembership`, normal generation stays structure-unit-native via `generation_key` and `anchor_unit`, and display/grouping uses `anchor_unit` and audience rows. The guarded mirror cleanup command (`cleanup_bible_study_v2_small_group_mirrors`), the one-time mirror→audience backfill (`backfill_bible_study_meeting_audience_scopes`), and the legacy-vs-membership shadow audit (`audit_bible_study_membership_readiness`) were retired with it. This did not remove the `SmallGroup` table, V1 `BibleStudySession`, `BibleStudyMeetingAudienceScope`, `anchor_unit`, or `generation_key`.
- **Legacy roles/capabilities and TeamAssignment / My Serving / ministry team membership** — kept separate; serving is never inferred from church-structure membership.

Already-switched consumers (ServiceEvent audience rows, Prayer group requests, Bible Study v2 meeting visibility / Today / role-worship pickers, group-progress roster and permission-fenced default group, the group-progress permission / accessible group list, and ordinary-member group reflection read **and write** paths) are listed in the next section.

## Non-Consumers / Do Not Migrate Casually

Do not migrate these casually as part of small-group cleanup:

- Permissions / capabilities: `ChurchRoleAssignment` and `accounts.permissions` are a role/capability system, not a membership-core visibility consumer.
- TeamAssignment / My Serving: serving assignments use `TeamMembership` and `TeamAssignmentMember`; do not infer serving roles from `ChurchStructureMembership`.
- MinistryTeam membership: ministry team membership remains a separate operational model.
- `BibleStudyMeetingRole`: treat as meeting-specific assignment metadata, not a ministry serving assignment or permission model.
- Reading progress and group progress: high privacy and permission risk; requires separate planning. CS-CORE.4F.1 switched only the group-progress visible roster (`member_rows`) to membership-core, and CS-CORE.4F.2 switched only the no-`?group=` default selected group to a permission-fenced membership-core candidate (used only when already inside the legacy-accessible list), both after zero-drift gate evidence; the group-progress permission and the accessible group list then switched to structure-aware role scopes plus membership-core own-group resolution in CS-CORE.2D-B (foundation CS-CORE.2D-A), so `Profile.small_group` no longer grants group-progress access.
- Reflection/comment privacy: high privacy risk.
